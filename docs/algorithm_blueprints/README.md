# Algorithm Blueprints (External)

This directory contains externally injected high-level algorithm blueprints.
These documents are **design inputs**, not evidence.

## Scope and role

- These blueprints describe proposed or planned algorithm systems.
- They **cannot** support or upgrade claims by themselves.
- Implementing a blueprint and producing evidence packages is a separate
  responsibility, governed by the research graph workflow.

## Priority order

1. **Phase 1 — OBGHI core algorithm** (`phase_01_OBGHI_core_algorithm.md`)
   Current focus. Implement the smallest runnable slice, produce evidence
   packages with metrics and failure modes.

2. **Phase 2 — R-MF-CTAS research system**
   (`phase_02_R-MF-CTAS_research_system.md`)
   Only pursue after Phase 1 has runnable, audited evidence.

3. **Phase 3 — Q-R-MF-CTAS observation physics**
   (`phase_03_Q-R-MF-CTAS_observation_physics.md`)
   Only when real, multi-height, or NV observation data (or observation-design
   work) is available.

4. **Phase 4 — EV-FAEDA industrial decision system**
   (`phase_04_EV-FAEDA_industrial_decision_system.md`)
   Industrial FA/EDA integration planning only.

## Usage rules

- Do not treat blueprint text as experimental evidence.
- Do not update claim status from blueprint text alone.
- Implement the smallest runnable algorithmic slice first.
- If blueprint theory conflicts with evidence metrics, evidence metrics win.
- Any implemented blueprint work must become a normal evidence package with
  metrics, RUN_REPORT, failure modes, cannot_claim, and research graph updates.
