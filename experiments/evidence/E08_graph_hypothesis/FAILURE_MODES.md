# Failure Modes

## 1. H3 over-explains everything

If complexity/L1 penalties are too small, the artifact hypothesis may explain true-via or return-path cases by adding a cheap local basis. This is why complexity is selected on validation and all baseline scores are reported.

## 2. H1 via still wins on return-path cases

A return path can generate swirl-like local residuals. If the return candidate library is too weak, the graph scorer may still prefer H1. This indicates missing return-current bases, not merely a threshold problem.

## 3. Sheet-only fit absorbs weak via current

If the via current is weak relative to sheet currents and noise, H0 may win. This is an information-limit regime; the recommended behavior is uncertainty/refusal rather than forced H1.

## 4. Synthetic graph too simple

This experiment is a controlled prototype. It must not be interpreted as real CAD/QDM validation. The next upgrade should use PyPEEC/CAD graph-labeled cases.

## 5. Metric over-polishing

`all_scientific_gates_passed` is not a claim that the real task is solved. It only says the graph-identification prototype gives a nontrivial, leakage-controlled comparison against raw and residual via baselines.

## 6. OOD no-via can be misread as return current

The current OOD failure table shows that the dominant residual error is not a
false high-confidence via alarm. Several no-via cases are classified as
`H2_sheet_return`, because an added return basis can reduce the field residual
under stronger OOD geometry, current-scale, and background-return shifts. This
is safer than hallucinating a via, but it still means the generated candidate
library is not a full physical explanation of all no-via ambiguity.

## 7. Weak or geometry-shifted true vias can be absorbed by H0/H2

Some OOD true-via cases are predicted as sheet-only or return-current cases
with very small residual margins. This is the information-limit regime that
motivates selective-risk reporting: a usable diagnostic system should report
low confidence or ask for more evidence rather than force a high-confidence
via/no-via decision.

## 8. Real layout import is still missing

Exp08's graph is generated by the experiment. It demonstrates the value of
graph-level hypothesis scoring, but it does not yet parse Gerber/GDS/ODB++,
does not know real pads/ports/planes, and does not solve layout-constrained
current flow on a measured circuit.

## 9. PyPEEC fields are much harder than centerline bridge fields

The exp07 bridge intentionally evaluates the same graph scorer on `B_centerline`
and `B_pypeec`. The centerline bridge is internally consistent, but PyPEEC
fields include voxel/body-current and finite-width effects that the current
centerline graph bases cannot fully explain. A large accuracy drop is expected
and should be reported as solver/operator gap evidence, not hidden.

## 10. Hidden mechanisms expose incomplete hypothesis libraries

When the true field contains a shifted via, hidden return, mismatched artifact,
or combined via+return mechanism, the current one-extra-component hypotheses can
fail sharply. This is the point of P1: the graph framing is better than pixel
thresholding, but it still needs a richer model bank before it can be trusted
on real circuits.

## 11. Via marginalization fixes registration, not physics mismatch

The fixed offset-grid search can recover shifted-via hidden stress because that
failure is mostly a candidate-registration error. It does not automatically
improve the exp07 PyPEEC bridge. If the PyPEEC field differs because of
finite-width/body-current physics or missing return/artifact bases, moving the
via candidate can lower interpretability and may hurt AUC.

## 12. Unknown rejection can become too conservative

The current unknown/out-of-library gate is calibrated on clean validation
residual and margin statistics. It correctly rejects hidden-mechanism stress,
but this is a conservative first pass. A practical system must eventually
separate "safe refusal" from "useful high-confidence diagnosis" with
risk-coverage curves on real or held-out high-fidelity data.

## 13. PyPEEC-aware basis banks can improve fit while hurting classification

The fixed finite-width, return-bank, artifact-bank, distributed-via, and
combined PyPEEC-aware basis variants are diagnostic model-bank probes. They can
lower the PyPEEC residual; for example, the combined basis reduces median
PyPEEC residual, while finite-width sheet bases improve PyPEEC bridge
classification. However, over-complete bases can also make the wrong
hypothesis easier to fit. A lower magnetic residual is therefore not by itself
proof that the graph explanation is physically correct.

## 14. Global registration is separable from solver physics

The global graph-to-field search checks translation, rotation, and scale
uncertainty separately from local via-position marginalization. If it improves
PyPEEC bridge residual only slightly, the remaining gap should be treated as
operator/basis mismatch rather than hidden coordinate misalignment.

## 15. Risk-coverage can reveal unusable refusal behavior

Unknown rejection should not be judged only by a single hidden-stress rejection
rate. The risk-coverage table reports what happens as coverage increases. A
useful diagnostic should preserve high accuracy on accepted clean in-library
cases and expose hidden/out-of-library cases as high risk; otherwise it may be
honest but operationally unhelpful.

## 16. H0/no-via remains the hardest practical class

PyPEEC no-via rows are often over-explained as return or artifact hypotheses.
H0-conservative evidence can protect no-via cases, but it may reduce true-via
recall. This is not a threshold nuisance; it is the core model-selection
trade-off that any reliable diagnostic must solve.

## 17. Unknown signals can select plausible but wrong hidden cases

Unknown detector ablations show that margin, residual, registration
instability, and combined scores can reject many hidden rows at a fixed clean
false-reject rate. They still do not guarantee low accepted-hidden risk. This
means unknown detection needs more physical evidence, not just scalar
confidence.

## 18. Registration search has a finite capture range

The synthetic registration curve shows that fixed global search can recover
some translation, rotation, and scale errors, but only within the configured
search bank. Larger or more structured layout distortions will require a real
registration/calibration module.

## 19. Model-selection objectives can hide recall loss

The formal model-selection audit can rank an H0-conservative rule highest
because it protects no-via cases. That is valuable for safety, but it also
reduces H1/true-via accuracy relative to less conservative choices. A
high-ranked objective is therefore not a final detector; the H0/H1 trade-off
must be reported explicitly.

## 20. Unknown safety is not deployment readiness

`combined_unknown_score` currently satisfies the clean false-reject budget and
rejects many hidden-mechanism rows, but this is still a synthetic screen. It
does not prove reliable real-world refusal, and it does not replace held-out
PyPEEC/CAD/QDM validation.

## 21. Active-design utilities must respect feasibility

Some label-free policies can increase evidence separation only by using
unrealistic excitation changes. The constrained active-design table marks such
policies as `constraint_limited` or `not_allowed`. A high unconstrained utility
must not be treated as a hardware protocol.

## 22. Standoff and tilt are not ordinary xy registration

Global translation/rotation/scale search can recover some lateral registration
errors. Sensor-height shifts and tilt change the observation operator itself;
the current table shows that standoff can sharply reduce accuracy and is not
repaired by the xy/global transform bank. Future real-data work needs explicit
height/tilt calibration.

## 23. Held-out PyPEEC pilots can have high variance

The current PyPEEC mini distribution is large enough to test the split protocol
but still small enough that calibration ranking and held-out ranking can differ.
If one evidence rule wins held-out but not calibration, that is not a final
model-selection theorem. It is a warning that a larger PyPEEC/CAD/FEM split is
needed before deployment calibration claims.

## 24. Physical unknown signals can reject more hidden cases but still accept risk

Entropy, residual-gap ambiguity, H0/H1 ambiguity, and residual-vs-score
disagreement add more physical evidence than margin/residual alone. They can
increase hidden rejection, but accepted hidden accuracy can remain too low.
Therefore they are safety diagnostics, not an operational guarantee.

## 25. Single PyPEEC splits can still overstate model-selection confidence

The deterministic held-out split is useful, but the current PyPEEC bridge is
still small. Repeated stratified splits show that the H0-conservative finite-
width rule is selected most often, while less conservative rules can preserve
more H1 recall. This means model-selection claims should report ranking
stability, not only the best single split.

## 26. Balanced PyPEEC mini distributions are still not final calibration data

The bridge now reaches the configured H0/H1/H2/H3 target of 100 cases per
hypothesis, including explicit H3 bend-artifact rows. This removes the previous
missing-H3 hard gap, but the cases are still generated toy PyPEEC geometries. Any
PyPEEC calibration or held-out model-selection result must remain a mini-
distribution pilot until external CAD/FEM/QDM or real-layout PyPEEC data expands
the distribution.

## 27. Hidden rejection rate is not the same as hidden safety

A signal can reject many hidden mechanisms while the hidden cases that remain
accepted are still frequently wrong. Accepted-hidden risk is therefore a
primary safety endpoint. A high rejection rate alone should not be treated as a
deployable unknown/refusal mechanism.

## 28. H0 safety can be improved without solving H1 recall

Class-specific margin refusal can create a safer trusted-output subset for
PyPEEC no-via/H0 rows, but that is not equivalent to a reliable via detector.
The key endpoint is not only H0 false-any rate; it is the joint behavior of H0
safety, accepted accuracy, coverage, and H1/true-via acceptance. If a policy
protects no-via rows by rejecting most true-via rows, the system has become
more honest but not yet more capable. This is the current core bottleneck under
single-state, no-CAD PyPEEC stress.

## 29. Evidence fusion can beat hand-written rules, but changes the claim type

The stacked evidence calibrator shows that the current PyPEEC basis/evidence
bank contains much more discriminative information than any single hand-written
rule exposes. However, it uses PyPEEC calibration folds, so it is no longer a
frozen no-calibration bridge result. Treat it as a calibration/held-out
breakthrough candidate: it can support a future deployable evidence model only
after the calibration/test split is made larger and more external.

## 30. Group-heldout and hidden-mechanism stress remain the hard tests

The stacked calibrator transfers well across variant-mod heldout folds and
operator-shifted `B_finite`/`B_centerline` fields, but family leaveout and
hidden mechanisms remain difficult. This means the current breakthrough is real
inside the balanced generated PyPEEC distribution, while out-of-family and
out-of-library safety still require stronger CAD-like distributions, explicit
unknown rejection, or additional observations.

## 31. Feature ablation is required before interpreting the calibrator

High accuracy alone does not explain why the stacked calibrator works. The
feature ablation shows that basis/evidence decisions and H0-conservative
cross-basis features are important, while some score-only views still preserve
H0 safety by sacrificing H1 recall. Therefore the method should be described as
calibrated evidence fusion, not as a black-box replacement for physical
hypothesis scoring.

## 32. Feature-distance OOD rejection is a safety screen, not a new class

Confidence-margin rejection remains unsafe on hidden mechanisms because a hidden
case can still receive a confident but wrong in-library label. The stacked
feature-distance screen asks a different question: does this evidence vector
look close to any calibrated H0/H1/H2/H3 class at all? In the current synthetic
hidden stress it rejects all hidden rows while preserving high accepted-clean
accuracy, which is a material mitigation breakthrough. The boundary is equally
important: rejected hidden rows are not explained or repaired, and this result
does not prove real CAD/FEM/QDM unknown safety.

## 33. Refusing unseen families is safer than pretending to generalize

Pure family leaveout remains a hard test for the stacked calibrator. The
feature-distance refusal audit shows that many unseen generated families are
rejected rather than accepted with wrong labels. This is a safety improvement,
not a cross-family classification solution. A deployable CAD-like system needs
either broader family coverage, explicit graph/layout priors, or a reporting
mode that distinguishes "known class" from "out-of-library family".

## 34. Near-boundary hidden cases are the real unknown-safety target

Base hidden mechanisms can be far from the in-library evidence manifold.
Near-boundary hidden cases deliberately sit closer to known return, via, or
bend/artifact candidates. If the distance screen only worked on far outliers it
would not be enough. The current near-boundary table shows a lower rejection
rate than base hidden stress, which is expected, while accepted near-boundary
rows preserve the primary label in this synthetic setting. Future work should
keep hardening these near-manifold unknowns before any real deployment claim.

## 35. Few-shot family adaptation is not the same as external generalization

The family-heldout failure is not purely information-theoretic inside the
generated PyPEEC distribution: a few family-specific calibration rows can turn
mostly-refused families into high-accuracy accepted diagnoses. This is a major
adaptation signal. It still does not prove zero-shot CAD/FEM/QDM generalization;
it says a practical system should plan for a small calibration phase whenever a
new board family or layout family is introduced.

## 36. Primary-label correctness can hide mechanism-level errors

Accepted near-boundary hidden rows can be correct at the coarse H0/H1/H2/H3
level while still having an unexplained hidden mechanism, such as a wrong-layer
or shifted via. Accepted-case audits therefore report mechanism status
separately. Deployment claims should not collapse primary-label correctness,
mechanism-level correctness, and parameter-level correctness into one number.
